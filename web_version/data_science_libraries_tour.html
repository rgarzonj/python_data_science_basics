pandas — the data wrangler

"80% of data science is cleaning data. Pandas is what you use to do it."

What it is: A library for working with tabular data (think: spreadsheets, but programmable). The core object is the DataFrame — a table with named columns and an index. Almost every data science project starts and ends with pandas.

▶In [1]
import pandas as pd
import numpy as np

# Generate the insurance dataset we will reuse throughout
np.random.seed(42)
n = 1500
df = pd.DataFrame({
    "policy_id": range(10001, 10001 + n),
    "age":      np.random.randint(18, 75, n),
    "bmi":      np.clip(np.random.normal(28, 6, n), 16, 50).round(1),
    "children": np.random.choice([0, 1, 2, 3, 4], n, p=[.42, .24, .20, .10, .04]),
    "smoker":   np.random.choice(["yes", "no"], n, p=[.20, .80]),
    "region":   np.random.choice(["NE", "SE", "SW", "NW"], n),
    "claims_filed": np.random.poisson(0.7, n),
})
df["annual_charges"] = (
    250 * df["age"]
    + 350 * (df["bmi"] - 25).clip(lower=0)
    + 22000 * (df["smoker"] == "yes").astype(int)
    + 500 * df["children"]
    + np.random.normal(0, 2500, n)
).clip(lower=1000).round(2)

print(f"DataFrame shape: {df.shape}")
df.head(8)

The five operations you will use every day

These five patterns cover the majority of real-world pandas work.

▶In [2]
# 1. FILTERING — pull rows that match a condition
high_risk = df[(df["smoker"] == "yes") & (df["bmi"] > 30)]
print(f"1. Filter:  {len(high_risk)} high-risk customers (smokers with BMI > 30)")

# 2. GROUPBY — split-apply-combine, the heart of analysis
by_region = df.groupby("region").agg(
    customers=("policy_id", "count"),
    avg_charges=("annual_charges", "mean"),
    total_claims=("claims_filed", "sum"),
).round(0)
print("\n2. Group by region:")
print(by_region)

# 3. CREATING NEW COLUMNS — feature engineering
df["risk_tier"] = pd.cut(df["annual_charges"],
                          bins=[0, 10000, 25000, 100000],
                          labels=["Preferred", "Standard", "High-Risk"])
print("3. New column — risk tier distribution:")
print(df["risk_tier"].value_counts())

# 4. PIVOT TABLES — Excel pivot tables, but programmable
pivot = df.pivot_table(values="annual_charges",
                       index="region", columns="smoker",
                       aggfunc="mean").round(0)
print("\n4. Pivot — avg charges by region × smoker:")
print(pivot)

Mental model

If you can do it in Excel — sort, filter, pivot, vlookup, formulas — you can do it 1000× faster on 100× more data with pandas. Everything else in this notebook builds on top of pandas.

matplotlib + seaborn — static visualisation

Matplotlib is the foundational plotting library — flexible but verbose. Seaborn is built on top of matplotlib and provides a higher-level, statistics-aware API with prettier defaults. Most teams use seaborn for quick exploration and drop down to matplotlib for fine-grained customisation.

▶In [5]
import matplotlib.pyplot as plt
import seaborn as sns

sns.set_theme(style="whitegrid")
plt.rcParams["figure.figsize"] = (10, 4)

fig, axes = plt.subplots(1, 3, figsize=(14, 4.2))

# Histogram — distribution of one variable, split by group
sns.histplot(data=df, x="annual_charges", hue="smoker", bins=40,
             ax=axes[0], palette=["#5d6e3a", "#c2410c"])
axes[0].set_title("Distribution: charges by smoker")

# Scatter — relationship between two variables
sns.scatterplot(data=df, x="age", y="annual_charges", hue="smoker",
                ax=axes[1], alpha=0.5, palette=["#5d6e3a", "#c2410c"], s=22)
axes[1].set_title("Age vs charges")

# Box plot — distribution across categories
sns.boxplot(data=df, x="region", y="annual_charges", ax=axes[2],
            hue="region", legend=False, palette="Set2")
axes[2].set_title("Charges by region")

plt.tight_layout()
plt.show()

When to use which
  • Seaborn — exploring data, quick statistical plots, presentations
  • Matplotlib — when you need pixel-perfect control (publication figures, custom annotations)
  • Both together — start in seaborn, refine the result with matplotlib commands

plotly — interactive visualisation

Where matplotlib and seaborn produce static images, Plotly produces interactive HTML charts you can zoom, pan, and hover. It is the go-to library for dashboards and shareable analyses. Plotly is not pre-built into Pyodide, so we describe its code rather than run it — but the API is straightforward.

When to use Plotly
  • Dashboards & web apps — Plotly powers Streamlit, Dash, and many internal tools
  • Sharing analyses — embed in HTML emails, Confluence, Notion
  • Multi-dimensional data — 3D plots, sunbursts, sankey diagrams

For static reports and academic papers, stick with matplotlib/seaborn.

scikit-learn — classical machine learning

The single most important ML library in Python. It implements virtually every classical algorithm — regression, classification, clustering — behind a consistent API. Once you have learned the pattern for one algorithm, you have learned it for all of them: model = SomeModel() → model.fit(X, y) → predictions = model.predict(X_new).

▶In [8]
from sklearn.model_selection import train_test_split, cross_val_score
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.pipeline import Pipeline
from sklearn.metrics import classification_report, roc_auc_score

# Business problem: predict which customers will file a claim this year
df["will_claim"] = (df["claims_filed"] > 0).astype(int)
X = pd.get_dummies(df[["age", "bmi", "children", "smoker", "region"]], drop_first=True)
y = df["will_claim"]

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, random_state=42)
print(f"Train: {len(X_train)} customers  ·  Test: {len(X_test)} customers")
print(f"Base rate (anyone files a claim): {y.mean():.1%}")
▶In [9]
# Pipelines — sklearn's killer feature.
# Bundle preprocessing + model so they travel together (no leakage, easy to deploy)
pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("model",  GradientBoostingClassifier(random_state=42)),
])
pipe.fit(X_train, y_train)

# Cross-validation: train/test on 5 splits, report stability
cv_auc = cross_val_score(pipe, X_train, y_train, cv=5, scoring="roc_auc")
print(f"5-fold CV AUC:    {cv_auc.mean():.3f} ± {cv_auc.std():.3f}")

# Held-out test performance
test_auc = roc_auc_score(y_test, pipe.predict_proba(X_test)[:, 1])
print(f"Held-out test AUC: {test_auc:.3f}\n")
print(classification_report(y_test, pipe.predict(X_test)))
Why scikit-learn dominates
  • Consistent API across hundreds of algorithms
  • Pipelines prevent the most common ML bug: data leakage
  • Battle-tested — in production at thousands of companies for over a decade

If your problem fits in a pandas DataFrame, scikit-learn is almost always the right starting point — even before reaching for deep learning.

statsmodels — statistical inference

Scikit-learn answers "what will happen?" — it is optimised for prediction. Statsmodels answers "why does it happen, and how confident are we?" — it is optimised for inference. If you need p-values, confidence intervals, and standard errors (regulatory filings, A/B tests, peer-reviewed papers), this is the library.

▶In [10]
import statsmodels.formula.api as smf

# R-style formula syntax — incredibly readable
model = smf.ols("annual_charges ~ age + bmi + children + C(smoker) + C(region)",
                data=df).fit()
print(model.summary().tables[1])

How to read this: coef is the effect on charges per unit of the variable, holding others constant. std err is the uncertainty. P>|t| is the p-value — probability the effect is actually zero. The 95% confidence interval is on the right.

For example, C(smoker)[T.yes] having a coefficient near +22,000 with p < 0.001 means: "Smokers cost roughly $22,000 more per year, and we are highly confident this effect is real."

When you need statsmodels over scikit-learn
  • Regulated pricing — regulators want p-values and confidence intervals
  • A/B test analysis — proper statistical tests with CIs
  • Time series — ARIMA, state space models, etc. are best-in-class
  • Causal inference — instrumental variables, fixed effects

pytorch — deep learning

The dominant framework for neural networks, used by OpenAI, Meta, Tesla, and most academic ML research.

When you actually need it: unstructured data (images, audio, text, video), very large datasets, or custom architectures. When you don't: for tabular data with under ~100k rows, scikit-learn (especially gradient boosting) usually beats deep learning while being simpler and faster.

PyTorch does not run in a browser, so we show the code rather than execute it. The pattern below — a four-line training loop — is the heart of all deep learning.

▶In [11]
# This cell is illustrative — PyTorch is too large for the browser runtime.
# Run it in Colab or any Python environment with `pip install torch`.

import torch
import torch.nn as nn

# A simple network: predicts claim probability from policy features
class ClaimPredictor(nn.Module):
    def __init__(self, n_features):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(n_features, 32), nn.ReLU(), nn.Dropout(0.2),
            nn.Linear(32, 16),         nn.ReLU(),
            nn.Linear(16, 1),          nn.Sigmoid(),
        )
    def forward(self, x):
        return self.net(x)

model     = ClaimPredictor(n_features=10)
optimizer = torch.optim.Adam(model.parameters(), lr=0.01)
loss_fn   = nn.BCELoss()

# THE CANONICAL TRAINING LOOP — the 4 lines at the heart of all deep learning
for epoch in range(100):
    optimizer.zero_grad()           # 1. reset gradients
    preds = model(X_train_t)        # 2. forward pass
    loss  = loss_fn(preds, y_train_t)
    loss.backward()                 # 3. backprop — compute gradients
    optimizer.step()                # 4. update weights
The four PyTorch concepts you must understand
  1. Tensors — n-dimensional arrays that can run on GPUs
  2. nn.Module — the base class for any neural network
  3. Autograd — automatic differentiation (PyTorch tracks gradients for you)
  4. The training loop — forward → loss → backward → step

Everything else in deep learning is a variation on this theme.

Hugging Face transformers — pretrained models

Training a deep learning model from scratch is expensive. The 2020s breakthrough was that you almost never need to — you can download a model someone else already trained on billions of examples, and either use it directly or fine-tune it on your problem. Hugging Face is the GitHub of pretrained models, hosting over 800,000 models behind a unified API.

▶In [12]

from transformers import pipeline

# Sentiment analysis on customer feedback
sentiment = pipeline("sentiment-analysis")

customer_feedback = [
    "The claims process was incredibly smooth and the agent was helpful.",
    "I have been waiting three weeks for a response. This is unacceptable.",
    "Premium increased without explanation. Considering switching providers.",
    "Quick payout after the accident — very impressed with the service.",
]
results = sentiment(customer_feedback)
for text, result in zip(customer_feedback, results):
    print(f"[{result['label']:8s} {result['score']:.2f}]  {text}")

#   pipeline("summarization")             # summarise long claim narratives

Why this changed the industry

Five years ago, building a sentiment classifier required a team of ML engineers, weeks of work, and labelled data. Today, the cell above does it in three lines, with better accuracy than that team could have achieved. Pretrained models are the single biggest leverage point in modern AI.

FastAPI — serving models as APIs

A trained model on your laptop creates zero business value. To matter, it must be callable from your other systems — the website, the underwriting tool, the claims platform. FastAPI wraps Python code as a production-grade web API in a few lines.

▶In [13]
# Save as `serve.py` and run: uvicorn serve:app --reload
# Any system on your network can then POST customer features and get a prediction.

from fastapi import FastAPI
from pydantic import BaseModel
import joblib

model = joblib.load("claim_model.pkl")
app = FastAPI(title="Insurance Claim Predictor")

class CustomerProfile(BaseModel):
    age: int
    bmi: float
    children: int
    smoker: str   # "yes" or "no"
    region: str

@app.post("/predict")
def predict_claim(profile: CustomerProfile):
    features = [[profile.age, profile.bmi, profile.children,
                 1 if profile.smoker == "yes" else 0]]
    probability = model.predict_proba(features)[0, 1]
    return {
        "claim_probability": round(float(probability), 3),
        "risk_tier":         "high" if probability > 0.5 else "low",
        "model_version":     "v1.0",
    }

What this gives you for free: automatic input validation (the CustomerProfile schema), auto-generated interactive docs at /docs, JSON serialisation, async support, production-grade performance.

The model lifecycle in production
  1. Train the model in a notebook (sklearn, pytorch)
  2. Save it to disk (joblib.dump or torch.save)
  3. Wrap it with FastAPI
  4. Deploy to a server (Docker → Kubernetes / Cloud Run / SageMaker)
  5. Monitor — track input distributions, prediction quality, latency

Steps 4 and 5 — sometimes called MLOps — are where most projects struggle.

Streamlit — internal apps without a frontend team

FastAPI exposes a model to other systems. Streamlit exposes a model to other humans — without writing a line of HTML, CSS, or JavaScript. You write a Python script. Streamlit turns it into a clickable web app.

▶In [14]
# Save as `app.py` and run: streamlit run app.py

import streamlit as st
import pandas as pd
import joblib
import plotly.express as px

st.set_page_config(page_title="Insurance Risk Explorer", layout="wide")
st.title("🏥 Insurance Risk Explorer")
st.markdown("Internal tool for underwriters — quote a customer in seconds.")

model = joblib.load("claim_model.pkl")
df    = pd.read_parquet("policies.parquet")

# Sidebar — interactive inputs
st.sidebar.header("Customer profile")
age      = st.sidebar.slider("Age", 18, 75, 40)
bmi      = st.sidebar.slider("BMI", 16.0, 50.0, 27.0, step=0.5)
children = st.sidebar.slider("Children", 0, 5, 1)
smoker   = st.sidebar.selectbox("Smoker", ["no", "yes"])

# Live prediction
features = [[age, bmi, children, 1 if smoker == "yes" else 0]]
prob = model.predict_proba(features)[0, 1]

col1, col2, col3 = st.columns(3)
col1.metric("Claim probability", f"{prob:.1%}")
col2.metric("Risk tier",         "High" if prob > 0.5 else "Low")
col3.metric("Suggested premium", f"${5000 + prob*30000:,.0f}")

st.subheader("How this customer compares to the book")
fig = px.scatter(df, x="age", y="annual_charges", color="smoker", opacity=0.4)
fig.add_scatter(x=[age], y=[prob*30000], mode="markers",
                marker=dict(size=20, color="red", symbol="star"),
                name="This customer")
st.plotly_chart(fig, use_container_width=True)

# That's the entire app. Run with: streamlit run app.py

A practical learning order

  1. pandas — invest serious time here, it pays off forever
  2. matplotlib + seaborn — enough to explore data confidently
  3. scikit-learn — your default for any predictive problem
  4. statsmodels — when you need rigorous statistics
  5. plotly + streamlit — when you need to share results
  6. FastAPI — when something is going to production
  7. PyTorch + Hugging Face — when tabular methods are not enough

Most data scientists are paid mostly for what they do with the first three. The rest are leverage on top of that foundation.
